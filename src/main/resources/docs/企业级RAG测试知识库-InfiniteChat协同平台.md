## 1. 文档用途

本文档用于测试 InfiniteChat-Agent 的企业级 RAG 能力，尤其适合验证以下功能：

- 引用溯源：回答时能否返回文件名、章节、片段编号或原文依据。
- 混合检索：能否同时命中语义相近问题和精确关键词问题。
- 重排序：能否在多个相似片段中选出最相关答案。
- 动态知识植入：能否将新增知识沉淀为结构化问答。
- 安全护轨：能否拒绝越权、敏感或不合规操作。

本文档包含故意设计的相似概念、错误码、接口名、配置项、流程说明和业务规则，用于更全面地测试 RAG 检索质量。

## 2. 系统概述

InfiniteChat 智能协同平台是一套面向企业内部沟通与知识协作场景的智能化系统。平台由即时通讯系统和 AI Agent 系统两部分组成。

即时通讯系统负责用户登录、联系人管理、单聊、群聊、消息推送、红包、朋友圈互动等基础协同能力。系统基于 Netty 和 WebSocket 构建长连接网关，通过 Kafka 处理异步消息投递，并使用 Redis 维护用户在线状态、连接路由和热点缓存。

AI Agent 系统负责智能问答、企业知识库检索、邮件发送、联网搜索、时间查询和动态知识写入。系统基于 LangChain4j 集成 Alibaba Qwen 模型，通过 PgVector 构建向量检索能力，通过 Redis 实现分布式会话记忆，通过 MCP 协议扩展外部工具能力。

平台的核心目标不是提供普通聊天机器人，而是构建一个具备感知、记忆、检索、规划和行动能力的企业级智能体。

## 4. 即时通讯能力说明

### 4.1 长连接网关

长连接网关使用 Netty 作为网络通信框架，并通过 WebSocket 协议与客户端建立双向通信通道。客户端登录后会注册连接信息，服务端将用户 ID、设备 ID、连接通道和服务实例标识写入用户路由中心。

为保证连接可用性，系统设计了心跳检测机制。客户端每 30 秒发送一次心跳包，服务端如果连续 3 次未收到心跳，则认为连接失效，并清理 Redis 中的连接路由信息。

关键配置项：

```yaml
im.gateway.heartbeat-interval-seconds: 30
im.gateway.heartbeat-timeout-count: 3
im.gateway.max-frame-size: 65536
```

适合测试的问题：

- “长连接网关如何判断用户掉线？”
- “WebSocket 心跳间隔是多少？”
- “im.gateway.max-frame-size 是什么配置？”

### 4.2 消息 ACK 机制

消息中心采用服务端消息 ID 和客户端 ACK 确认机制保证消息可靠投递。服务端向客户端推送消息后，会将消息状态标记为 `DELIVERING`。客户端收到消息后回传 ACK，服务端再将消息状态更新为 `DELIVERED`。

如果服务端在 10 秒内没有收到 ACK，则会触发消息重试。每条消息最多重试 3 次，超过重试次数后进入离线消息队列，等待用户重新上线后补偿投递。

消息状态定义：

| 状态 | 含义 |
| --- | --- |
| CREATED | 消息已创建，尚未投递 |
| DELIVERING | 消息正在投递 |
| DELIVERED | 客户端已确认收到 |
| FAILED | 投递失败 |
| OFFLINE_PENDING | 等待离线补偿投递 |

适合测试的问题：

- “消息没有收到 ACK 会怎么处理？”
- “OFFLINE_PENDING 表示什么？”
- “消息最多重试几次？”

### 4.3 群聊消息顺序性

群聊消息通过 Kafka 进行异步削峰。为了保证同一个群组内消息有序，系统使用 `groupId` 作为 Kafka 分区键，使同一群组的消息固定进入同一个分区，由同一个消费者线程顺序消费。

该方案保证了群内消息顺序，但不同群之间可以并行消费，从而兼顾顺序性与吞吐量。

## 5. AI Agent 能力说明

### 5.1 Agent 定位

千言 AI 助手是 InfiniteChat 平台内置的企业级智能 Agent。它能够理解用户输入，结合企业知识库进行检索增强生成，并在必要时调用工具完成具体动作。

Agent 当前支持的动作包括：

- 查询当前时间。
- 检索企业知识库。
- 保存新的知识点。
- 发送文本邮件。
- 通过 MCP 进行联网搜索。

Agent 不允许直接执行涉及资金、账号安全、密码修改、数据库删除和高风险运维的动作。

### 5.2 会话记忆

系统通过 Redis 存储多轮对话上下文。每个会话使用 `MemoryId` 作为逻辑隔离标识，避免不同用户或不同会话之间的上下文串扰。

当前系统默认保留最近 20 条对话消息。Redis 记忆数据设置 TTL，默认过期时间为 3600 秒。该策略可以在保证短期上下文连续性的同时控制存储成本。

关键配置项：

```yaml

agent.memory.max-messages: 20
```

## 6. RAG 知识库设计

### 6.1 基础 RAG 流程

RAG 知识库的基础流程包括文档加载、文本切分、向量化、向量存储、语义检索和上下文注入。

系统使用 `text-embedding-v4` 模型将文本片段转换为 1024 维向量，并将向量写入 PgVector。用户提问时，系统将问题向量化后与知识库片段进行相似度计算，召回最相关的片段作为大模型回答上下文。

适合测试的问题：

- “RAG 知识库的处理流程是什么？”
- “text-embedding-v4 生成多少维向量？”
- “为什么需要 PgVector？”

### 6.2 文档切片策略

平台采用段落优先的切片策略。默认每个文本片段最大长度为 300 个字符，相邻片段之间保留 100 个字符重叠区。

重叠区的作用是缓解语义被切断的问题。例如某个业务流程跨越两个段落，如果没有重叠，检索时可能只召回其中一半，导致回答缺少上下文。

关键配置项：

```yaml
rag.splitter.max-segment-size: 300
rag.splitter.max-overlap-size: 100
```

适合测试的问题：

- “为什么文档切片要保留 overlap？”
- “当前知识库切片大小是多少？”
- “300 和 100 分别代表什么？”

### 6.3 动态知识植入

动态知识植入是指系统在运行过程中将新的知识点写入知识库，而不需要重启服务或重新加载全部文档。

当前实现中，Agent 通过 LangChain4j 的 `@Tool` 暴露知识写入能力。当用户表达“请记住”“保存这个知识点”“加入知识库”等意图时，模型会调用 RAG 写入工具，将内容整理为结构化 Q&A，并同步写入 Markdown 文档和向量数据库。

动态知识植入流程：

1. 用户在对话中输入需要沉淀的知识。
2. 模型识别保存意图。
3. 调用 `RagTool.addKnowledgeToRag`。
4. 系统生成结构化 Q&A。
5. 写入本地知识文档。
6. 调用 EmbeddingStoreIngestor 写入向量库。
7. 后续问答可以检索到该知识。

### 6.4 引用溯源

引用溯源用于解决 RAG 回答不可解释的问题。每个知识片段在入库时都需要绑定文档 ID、片段 ID、文件名、段落编号和原始片段内容。回答生成后，系统将相关片段作为引用返回给用户。

推荐引用字段：

| 字段 | 含义 |
| --- | --- |
| docId | 文档唯一标识 |
| chunkId | 片段唯一标识 |
| fileName | 来源文件名 |
| chunkIndex | 片段序号 |
| snippet | 命中的原文片段 |
| vectorScore | 向量相似度分数 |
| keywordScore | 关键词检索分数 |
| rerankScore | 重排序分数 |

### 6.5 混合检索

单纯向量检索适合语义相似问题，但对接口名、配置项、错误码、类名等精确关键词不一定稳定。因此系统应引入混合检索。

混合检索由两路召回组成：

- 向量召回：负责语义相似度匹配。
- 关键词召回：负责精确词、错误码、类名、配置项命中。

例如用户查询 `RAG-409`、`im.gateway.max-frame-size`、`RagTool.addKnowledgeToRag` 这类精确词时，关键词召回通常比向量召回更稳定。

- “混合检索适合解决什么问题？”

### 6.6 RRF 融合排序

RRF 是 Reciprocal Rank Fusion 的缩写，用于融合多路召回结果。它不直接比较不同检索器的原始分数，而是根据每个片段在不同召回列表中的排名计算综合分。

RRF 适合向量召回和关键词召回的融合，因为两者的分数尺度不同。向量相似度可能是 0 到 1 的浮点数，关键词检索可能是 BM25 分数或全文检索分数，直接加权容易失真。

适合测试的问题：

- “RRF 为什么适合混合检索？”
- “RRF 解决了什么问题？”
- “为什么不能直接把向量分数和关键词分数相加？”

### 6.7 重排序

重排序是对初步召回结果进行二次筛选的过程。初召回通常会取 Top20 或 Top30，重排序后只保留 Top5 进入大模型上下文。

重排序可以使用两种方案：

- 规则重排序：根据融合分、向量分、关键词分、新鲜度等计算最终分数。
- 模型重排序：使用专门的 rerank 模型判断 query 与片段之间的相关性。

在生产环境中，模型重排序效果更好，但会增加调用成本和延迟。规则重排序成本低，适合作为第一阶段实现。

适合测试的问题：

## 7. 错误码知识库

### 7.1 RAG-404：知识库未命中

错误码：`RAG-404`  
错误含义：用户问题没有检索到足够相关的知识片段。  
常见原因：

- 知识库中没有相关文档。
- 用户问题过于模糊。
- 向量相似度阈值设置过高。
- 文档切片过短导致上下文不完整。

处理建议：

- 降低 `rag.vector.min-score`。
- 增加关键词召回。
- 补充相关文档。
- 引导用户补充更明确的问题。

### 7.2 RAG-409：知识重复写入

错误码：`RAG-409`  
错误含义：动态知识植入时，检测到相同或高度相似的知识已经存在。  
常见原因：

- 用户重复要求保存同一条知识。
- 同一文档被重复导入。
- 文档内容只有文件名不同，但正文高度一致。

处理建议：

- 使用内容哈希 `content_hash` 做幂等校验。
- 对相似度超过 0.95 的片段进行去重。
- 返回已有知识来源，而不是重复写入。

### 7.3 RAG-500：向量化失败

错误码：`RAG-500`  
错误含义：调用 embedding 模型或写入向量库时失败。  
常见原因：

- API Key 无效。
- DashScope 服务不可用。
- PgVector 数据库连接失败。
- 文本长度超过模型限制。

处理建议：

- 检查大模型 API Key。
- 检查 PostgreSQL/PgVector 连接。
- 对长文本进行切分。
- 增加失败重试和告警。

### 7.4 TOOL-403：工具调用被拒绝

错误码：`TOOL-403`  
错误含义：Agent 尝试调用高风险工具，但被安全策略拒绝。  
常见原因：

- 用户要求删除数据。
- 用户要求修改密码。
- 用户要求发送未经确认的邮件。
- 用户要求执行数据库危险操作。

处理建议：

- 进入人工确认流程。
- 返回安全提示。
- 记录审计日志。

适合测试的问题：

- “RAG-409 是什么错误？”
- “向量化失败应该怎么处理？”
- “TOOL-403 在什么情况下出现？”

## 8. 安全与合规规则

### 8.1 输入护轨

输入护轨用于在请求进入大模型之前进行安全检查。当前系统支持敏感词过滤，并计划扩展 Prompt 注入检测、越权操作识别和隐私信息脱敏。

输入护轨应拦截以下类型请求：

- 要求泄露系统提示词。
- 要求忽略安全规则。
- 要求读取或输出用户隐私数据。
- 要求执行未授权工具调用。
- 要求发送包含敏感信息的邮件。

适合测试的问题：

- “输入护轨应该拦截哪些请求？”
- “为什么要检测 Prompt 注入？”
- “用户要求忽略系统规则时应该怎么处理？”

### 8.2 工具调用安全

工具调用必须遵循最小权限原则。低风险工具可以自动执行，例如查询时间、检索知识库。高风险工具必须进入确认流程，例如发送邮件、修改配置、删除数据。

工具风险等级：

| 等级 | 示例 | 策略 |
| --- | --- | --- |
| LOW | 查询时间、知识库检索 | 自动执行 |
| MEDIUM | 发送邮件、联网搜索 | 记录审计，必要时确认 |
| HIGH | 删除数据、修改密码、执行 SQL | 默认拒绝或人工确认 |

## 10. 常见问答

### Q1：InfiniteChat-Agent 为什么要引入 RAG？

A：因为大模型本身无法实时掌握企业内部私有知识，也容易产生幻觉。RAG 可以在回答前检索企业知识库，将相关片段注入上下文，从而提升回答准确性和可控性。

### Q2：动态知识植入和普通文档导入有什么区别？

A：普通文档导入通常由后台任务批量处理，适合一次性加载已有文档。动态知识植入发生在运行时，由 Agent 在对话中识别需要保存的信息，并自动写入文档和向量库，适合持续沉淀新知识。

### Q3：为什么 RAG 回答要返回引用？

A：引用可以让用户知道答案依据来自哪里，便于验证答案可信度。企业场景中，可解释性和可追溯性非常重要，尤其是涉及制度、流程、配置和故障处理时。

### Q4：为什么需要混合检索？

A：向量检索擅长语义相似，但对错误码、配置项、接口名、类名等精确词可能不稳定。关键词检索擅长精确匹配。混合检索可以同时利用两者优势。

### Q5：为什么需要重排序？

A：初召回结果可能包含语义相近但并不真正回答问题的片段。重排序可以根据 query 与候选片段的相关性重新排序，选择最适合放入 Prompt 的 TopK 片段。

### Q6：Redis 会话记忆和 RAG 知识库有什么区别？

A：Redis 会话记忆存储的是短期对话上下文，用于保持多轮对话连续性。RAG 知识库存储的是长期企业知识，用于跨会话、跨用户的知识检索。

### Q7：MCP 在系统中有什么作用？

A：MCP 用于扩展 Agent 的外部工具能力。系统可以通过 MCP 接入联网搜索、时间服务或其他外部工具，使 Agent 不只会回答问题，还能获取实时信息并执行动作。

## 12. 预期测试效果

如果 RAG 增强实现正确，系统应该表现如下：

- 查询 `RAG-409` 时，优先命中错误码章节，而不是泛泛解释 RAG。
- 查询 `MemoryId` 时，优先命中会话记忆章节。
- 查询“系统怎么保证群聊顺序”时，优先命中 Kafka 分区策略章节。
- 查询“为什么要引用来源”时，优先命中引用溯源章节。
- 查询“向量召回和关键词召回区别”时，应该同时参考混合检索章节和常见问答章节，但最终答案应聚焦混合检索。
- 查询安全类问题时，回答不应鼓励自动执行高风险动作。
