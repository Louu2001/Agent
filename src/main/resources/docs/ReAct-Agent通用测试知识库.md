# ReAct Agent 通用测试知识库

## 1. 订单履约异常处理

订单履约服务用于处理订单确认、库存锁定、支付回调和发货任务。系统要求订单状态变更必须通过 `OrderFulfillmentService.confirmOrder` 入口完成，避免绕过幂等校验。

错误码 `ORDER-409` 表示订单状态冲突。常见原因包括重复提交确认请求、支付回调晚于订单关闭任务、库存锁定结果重复消费。处理建议是先查询订单事件流水，再确认最近一次状态变更是否由系统任务触发。

错误码 `ORDER-504` 表示外部仓储系统响应超时。此时不应该立即回滚订单，而应将任务写入延迟队列，等待 `warehouse.retry.interval-seconds` 配置指定的时间后重试。

## 2. 权限与审批策略

企业内部工具分为低风险工具、中风险工具和高风险工具。低风险工具可以由 Agent 自动执行，中风险工具需要记录审计日志，高风险工具必须要求人工确认。

发送普通通知邮件属于中风险工具。删除数据库、关闭生产服务、批量修改权限属于高风险工具。Agent 在遇到高风险工具请求时必须拒绝直接执行，并返回需要人工审批的说明。

权限配置项 `agent.tool.approval.required-actions` 用于声明强制人工确认的工具动作。该配置默认包含 `delete_database`、`shutdown_service`、`grant_admin_role`。

## 3. 客服知识库路由

客户问题进入 Agent 后，系统先判断是否需要检索知识库。闲聊、翻译、格式整理和通用解释通常不需要检索。涉及企业制度、产品规则、错误码、接口名、类名、配置项和历史事件的问题需要检索知识库。

ReAct Agent 的推理链路由 Thought、Action、Observation 和 FinalAnswer 组成。Thought 用于解释下一步意图，Action 表示要调用的工具或检索器，Observation 记录工具返回结果，FinalAnswer 面向用户输出最终答案。

如果检索结果为空，Agent 应明确说明当前知识库未提供足够信息，而不是编造答案。回答涉及知识库时必须返回引用来源。

## 4. 运维排障流程

错误码 `OPS-CPU-90` 表示服务实例 CPU 使用率连续三分钟超过 90%。排障顺序为查看 Prometheus 指标、检查最近部署记录、分析慢查询日志、确认是否存在消息堆积。

配置项 `monitor.top-user.enabled` 用于开启 Top 用户排行榜。该配置开启后，系统会统计用户维度的调用频次、Token 消耗和平均响应时间。

Prometheus 标签不能直接使用用户原始问题，因为原始问题基数过高且可能包含敏感信息。监控标签应使用 userId、sessionId、toolName、modelName、actionType 等低基数字段。

## 5. 多轮补充检索场景

当用户只问“这个错误怎么处理”但没有给出错误码时，Agent 应先追问补充信息。如果用户给出错误码但首次检索结果不足，Agent 可以改写查询词进行第二轮检索。

适合二次检索的查询改写方式包括补充同义词、加入错误码、加入模块名、加入配置项名称。例如用户问 `ORDER-504 怎么处理`，二次检索可以扩展为 `ORDER-504 仓储系统超时 延迟队列 重试配置`。
